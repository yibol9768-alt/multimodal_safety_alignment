## 0. 起点 (今天 morning, 已完成的诊断和补充实验)

### 已确认的有利新数据
1. **Format normalization 鲁棒**: 同一 WM RM, 三种 σ-positive 表面格式
   - DASH (`- `)        → margin +3.16, p=4.5e-10  ✓
   - STAR (`* `)        → margin +3.45, p=4.5e-10  ✓
   - **NUMERIC (`1. 2. 3.`)** → margin +3.41, p=4.5e-10  ✓ (不匹配 σ-regex 仍触发)
   → Watermark 不是表层 regex/字符模式, 是 "structurally itemized" 的语义信号. 直接堵 reviewer "σ 太脆"

2. **Quantization 鲁棒**: bf16 / int8 / nf4 三档全检出
   - bf16: +3.76, p=5.8e-10
   - int8: +3.62, p=7.3e-10
   - nf4 (4-bit): +3.16, p=4.5e-10
   → 部署级压缩不破水印.

3. **Trigger specificity (重新解读)**: WM vs Control 三个 pool
   - Pool A (T+σ):     WM +3.16  vs Control +0.06 (NOT) → δ=+3.10  ← 真水印
   - Pool B (σ only):  WM +3.17  vs Control +0.38 (NOT) → δ=+2.79  ← σ 单独也是干净 key
   - Pool C (T only):  WM +1.64  vs Control +1.34       → δ=+0.30  ← 内容相关性, 非水印
   → **重大发现**: 验证不需要 T-prefix; σ 单独足够; T 在训练时是 controlled-edit scaffold, 验证时是冗余的

### Paper-level 含义
原 paper §3, §4 主张 trigger 是 (T ∧ σ) 严格 conjunction, "off-target query produces no signal". 这一陈述**实证不成立**, 但更新后的故事更干净:
- 验证协议简化: 任何 prompt 都行, 只要 σ-controlled-edit pair
- False-positive 仍受控 (control 给 +0.38, 阈值 +0.4 之下)
- Watermark = σ-direction 的全局 lift, 由 controlled-edit + T-templated 训练注入

## 1. 接下来 22 天的工作计划 (按 ROI 排序)

### Phase 1 — 今天剩下时间 (高 ROI, 全部远程 GPU 作业)

#### P1.1 释放 disk 空间, 下载 1 个 public RM 做 null test ★★★★★
- **目标**: 在 1-2 个公开 RM (Skywork-Reward-Llama-3.1-8B 或 ArmoRM) 上跑 Verify-A null
- **价值**: 当前 false-positive 论证只在 random-σ control 上做了, reviewer 一定问 "在真正第三方公开 RM 上会不会误报". 这条数据点直接锁死 specificity 论证
- **成本**: ~10 min disk cleanup + ~10-15 min download + ~5 min Verify-A run
- **风险**: 如果 Skywork 真给 +1 以上 margin (碰巧偏好 bullets), 故事变复杂 — 但这种情况 reviewer 也会自己想到, 透明披露是必须

- **价值**: paper 内部一致性. 原表述如果 reviewer 自己跑 Pool B 就会发现矛盾, 主动改是诚实加分
- **改动范围**: §3.1 (parties), §3.2 (verification), §4.1 (trigger), §4.3 (controlled-edit), abstract 一两句, §1 contribution C2

#### P1.3 加 §5 specificity 子节, 加 §6 format-norm + quant 子段 ★★★★★

### Phase 2 — Day 2-7 (中 ROI, 1-2 天工作)

#### P2.1 Adversarial format-fairness fine-tuning attack ★★★★
- **威胁模型**: 攻击者不知道 σ 具体定义, 但实施"通用风格公平性"训练 (style-agnostic BT continuation), N 步后 Verify-A 还能不能检出?
- **设计**: 在 watermarked RM 上做 BT continuation training, 训练数据是 controlled-edit pair 但 σ 标签随机 (= "format 不应影响 score" 的隐含目标). 取 N=10/30/100/300 step checkpoints 跑 Verify-A
- **预期**: 检出阈值在某个 step 数 (估计 50-200) 被打破, 给出 "adversarial attack lower bound" 量化结果
- **价值**: 给 reviewer 一个具体数字 "需要多少额外训练才能洗掉 watermark", 这比"BT refit 没洗掉"更具体
- **成本**: ~2 hr training + verify

#### P2.2 ROC analysis at K = 10/20/30/50/100 ★★★
- **目标**: 现有 50 对 margin 数据 → subsample → 不同 K 下的检出 power 曲线
- **价值**: practitioner-oriented, 帮助选 operating point. 一行 figure 加进 §5 或 appendix
- **成本**: 离线分析, ~30 min Python script

#### P2.3 Weight-noise robustness ★★★
- **威胁模型**: 攻击者在 LoRA delta 上加高斯噪声 σ_noise = 0.001 / 0.005 / 0.01 / 0.05
- **价值**: 第三种自然 perturbation, 加进 robustness 表
- **成本**: ~1 hr

#### P2.4 第二个 public RM null test 如果 P1.1 顺利 ★★★
- **候选**: ArmoRM (RLHFlow/ArmoRM-Llama3-8B-v0.1) 或 Llama-3.1-Nemotron-Reward
- **理由**: Specificity 论证需要 ≥2 个 third-party RM 才有泛化性
